# 设计哲学对比：OpenSpec vs Superpowers

> 分析对象：[OpenSpec](https://github.com/Fission-AI/OpenSpec) (Fission-AI) 与 [Superpowers](https://github.com/obra/superpowers) (Jesse Vincent / Prime Radiant)

---

## 一、核心概念提炼

两个项目虽定位不同——OpenSpec 是规范驱动的变更管理工具，Superpowers 是编码代理的行为方法论——但底层共享四项设计原则：

### 1. "声明式优于命令式"

使用者只需声明期望的终态（OpenSpec 声明 artifact 依赖图、Superpowers 声明技能触发条件），框架负责推导执行路径。两者都不要求使用者编写命令式流程控制代码。

### 2. "状态即文件"

核心状态以文件系统中的文件形态存在：OpenSpec 的每个变更是一个包含 `proposal.md`、`specs/`、`design.md`、`tasks.md` 的文件夹；Superpowers 的每个技能是一个包含 `SKILL.md` 的文件夹。文件的存在与否本身就是状态标记。

### 3. "零依赖"

OpenSpec 的 Schema 解析优先从项目本地目录查找，允许完全脱离全局安装使用（`resolver.ts:44-46`）；Superpowers 明确拒绝第三方依赖引入核心（`CLAUDE.md:33`）。两者都追求最小外部耦合。

### 4. "行为塑造"

OpenSpec 通过 Schema 中 artifact 的 `requires` 和 `generates` 字段塑造 AI 代理的工作流；Superpowers 通过 SKILL.md 中的 Red Flags 表、合理化借口表和 CSO 描述塑造代理的行为边界。两者都不依赖硬编码逻辑，而是通过文档结构间接控制行为。

## 二、OpenSpec 做法

### 2.1 五大哲学原则

OpenSpec 在 README 中明确提出五条设计哲学：

```text
→ fluid not rigid
→ iterative not waterfall
→ easy not complex
→ built for brownfield not just greenfield
→ scalable from personal projects to enterprises
```

> 来源：`D:\AI-Agent\github-analyze\OpenSpec\README.md:28-34`

在 concepts.md 中对前四条有详细展开：

- **fluid not rigid**：传统规范系统将用户锁定在阶段中——先计划、再实现、再完成。OpenSpec 更灵活，用户可以按任意顺序创建工件（`concepts.md:17-18`）。
- **iterative not waterfall**：需求会变化，理解会加深。OpenSpec 拥抱这一现实（`concepts.md:20`）。
- **easy not complex**：一些规范框架需要大量设置、僵化格式或重量级流程。OpenSpec 秒级初始化，按需定制（`concepts.md:22`）。
- **brownfield-first**：大多数软件工作不是从零开始——而是修改已有系统。OpenSpec 的 delta-based 方法让变更描述成为一等公民（`concepts.md:24`）。

### 2.2 "依赖是使能者，不是门卫"

核心信条出现在 Schema 概念文档中：

> **Dependencies are enablers, not gates.** They show what's possible to create, not what you must create next. You can skip design if you don't need it. You can create specs before or after design — both depend only on proposal.

> 来源：`D:\AI-Agent\github-analyze\OpenSpec\docs\concepts.md:592`

这意味着 `requires` 字段表达的是"前置输入已就绪"而非"必须按序执行"。依赖图描述的是数据流向，不是控制流。用户可以跳过 `design.md`，可以在 `design` 之前或之后创建 `specs`——只要两者都依赖 `proposal`，它们就可以并行创建。

### 2.3 三层 Schema 解析优先级

`resolver.ts` 实现了三级 Schema 解析链，体现了"项目自主、用户可覆盖、包内提供默认"的分层治理理念：

```
1. Project-local: <projectRoot>/openspec/schemas/<name>/schema.yaml
2. User override: ${XDG_DATA_HOME}/openspec/schemas/<name>/schema.yaml
3. Package built-in: <package>/schemas/<name>/schema.yaml
```

> 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\resolver.ts:51-55`

实现代码在 `getSchemaDir` 函数中按此顺序依次检查文件是否存在（`resolver.ts:63-91`）。`listSchemasWithInfo` 函数进一步为每个 Schema 标注来源层级：`project`、`user`、`package`（`resolver.ts:224-229`）。

### 2.4 Artifact 流与 Delta Spec

OpenSpec 将变更流程建模为 artifact 依赖图（`concepts.md:358-366`）：

```
proposal → specs → design → tasks → implement
    │         │          │         │
   why      what       how      steps
 + scope   changes   approach  to take
```

Delta Spec 是其 brownfield-first 理念的核心实现——通过 ADDED / MODIFIED / REMOVED 三段式结构描述变更，避免全量重写规范（`concepts.md:489-524`）。

### 2.5 从规范到实现的完整闭环

归档（Archive）操作将 delta specs 合并进主规范，变更文件夹移入 `archive/`，实现规范演进的可追溯性（`concepts.md:643-693`）。这是"状态即文件"原则的集中体现：归档 = 文件合并 + 文件夹移动。

## 三、Superpowers 做法

### 3.1 四大核心理念

Superpowers 在 README 的 Philosophy 部分明确提出四条原则：

```text
- Test-Driven Development — Write tests first, always
- Systematic over ad-hoc — Process over guessing
- Complexity reduction — Simplicity as primary goal
- Evidence over claims — Verify before declaring success
```

> 来源：`D:\AI-Agent\github-analyze\superpowers\README.md:200-203`

### 3.2 "技能不是散文——是塑造代理行为的代码"

CLAUDE.md（贡献者指南）中明确将技能定义为行为控制手段：

> Skills are not prose — they are code that shapes agent behavior.

> 来源：`D:\AI-Agent\github-analyze\superpowers\CLAUDE.md:90`

这一声明确立了技能文档的双重身份：表面上是 Markdown 文档，实质上是行为约束代码。因此对技能内容的修改门槛极高——需要对抗性压力测试、前后评估对比（`CLAUDE.md:91-95`）。

### 3.3 "零依赖插件"的设计约束

Superpowers 对第三方依赖持零容忍态度：

> Superpowers is a zero-dependency plugin by design. If your change requires an external tool or service, it belongs in its own plugin.

> 来源：`D:\AI-Agent\github-analyze\superpowers\CLAUDE.md:33`

这一约束直接导致了可移植性优势：Superpowers 支持 Claude Code、Codex CLI、Codex App、Factory Droid、Gemini CLI、OpenCode、Cursor、GitHub Copilot CLI 等多种编码代理（`README.md:7-8`），因为不依赖任何特定运行时。

### 3.4 "你的人类伙伴"——术语即哲学

CLAUDE.md 中反复使用 "your human partner" 而非 "the user"，并明确声明这是刻意选择：

> Superpowers has its own tested philosophy about skill design, agent behavior shaping, and terminology (e.g., "your human partner" is deliberate, not interchangeable with "the user").

> 来源：`D:\AI-Agent\github-analyze\superpowers\CLAUDE.md:99`

这一术语选择暗示了 Superpowers 的核心立场：AI 代理是人类开发者的协作伙伴，不是工具。代理有责任保护人类伙伴免受尴尬——比如提交低质量 PR（`CLAUDE.md:9`）。

### 3.5 CSO 原则：描述只写触发条件

`writing-skills/SKILL.md` 提出了 Claude Search Optimization (CSO) 原则，其核心规则是：

> **CRITICAL: Description = When to Use, NOT What the Skill Does**
>
> The description should ONLY describe triggering conditions. Do NOT summarize the skill's process or workflow in the description.

> 来源：`D:\AI-Agent\github-analyze\superpowers\skills\writing-skills\SKILL.md:150-153`

这一原则源自实测发现：当描述中包含工作流摘要时，AI 代理会直接按描述执行而跳过完整技能内容。例如，描述中写 "code review between tasks" 导致代理只做一次审查，而技能流程图实际要求两阶段审查（`SKILL.md:155-158`）。因此描述必须只声明"何时触发"，让代理去阅读技能正文获取完整流程。

### 3.6 Anti-rationalization：系统化对抗合理化

SKILL.md 专门设立了"Bulletproofing Skills Against Rationalization"章节（`SKILL.md:460-523`），包含：

- **合理化借口表**：将代理在压力场景下使用的每一句借口记录在案（`SKILL.md:446-457`）
- **Red Flags 列表**：让代理在即将违规时自我检测（`SKILL.md:513-522`）
- **"字面违反即精神违反"原则**：在规则开头声明，截断"我遵循精神而非形式"类的辩解（`SKILL.md:490-493`）

这构成了一套完整的反合理化防御体系，将 TDD 的 RED-GREEN-REFACTOR 循环应用于文档：先观测代理在没有技能时的违规行为（RED），再编写针对具体借口的技能（GREEN），然后继续封堵新发现的漏洞（REFACTOR）（`SKILL.md:534-555`）。

### 3.7 Prompt 模板化与 SubAgent 分派

Superpowers 的 subagent-driven-development 模式将任务执行与审查分离。每个任务由独立子代理执行，通过两阶段审查（规范合规性审查 + 代码质量审查）保证质量（`README.md:162`）。技能文档中的流程图（flowchart）就是子代理的执行模板——代理读取流程图而非依赖自然语言描述。

## 四、对比分析

### 维度一：流程灵活性 vs 行为纪律性

| 维度 | OpenSpec | Superpowers |
|------|----------|-------------|
| 核心关切 | 给 AI 多大自由度 | 如何封堵 AI 的偏航 |
| 机制 | Schema DAG 允许跳过节点、并行创建 | Red Flags 表、合理化借口表封堵违规 |
| 关键引文 | "Dependencies are enablers, not gates"（`concepts.md:592`） | "Skills are not prose — they are code that shapes agent behavior"（`CLAUDE.md:90`） |

OpenSpec 信任 AI 的判断力，给予最大流程自由度——依赖图只表示"可以创建"而非"必须创建"。Superpowers 假设 AI 在压力下会走捷径，系统化地封堵每一条合理化路径。

### 维度二：工具化 vs 方法论化

| 维度 | OpenSpec | Superpowers |
|------|----------|-------------|
| 本质 | CLI 工具 + 文件结构约定 | 方法论 + 行为约束文档 |
| 安装 | `npm install -g`（`README.md:87`） | 复制 CLAUDE.md / 安装插件（`README.md:36-61`） |
| 运行时 | Node.js CLI 解析 YAML、管理文件夹 | 纯 Markdown，无运行时 |
| 扩展点 | 自定义 Schema（`concepts.md:617-636`） | 自定义 Skill（`SKILL.md` 全文） |

OpenSpec 是一个你可以 `npm install` 的工具；Superpowers 是一套你可以 `cp -r` 的方法论。前者依赖 TypeScript 解析器执行 Schema 约束，后者依赖 AI 代理阅读文档后自律执行。

### 维度三：扩展模式

| 维度 | OpenSpec | Superpowers |
|------|----------|-------------|
| 扩展方式 | Schema + 适配器（`resolver.ts:44-46`） | 零依赖保持可移植（`CLAUDE.md:33`） |
| 多工具支持 | 通过 `openspec update` 生成多种代理指令（`README.md:99`） | 直接支持 7+ 编码代理（`README.md:7-8`） |
| 层级覆盖 | 项目级 > 用户级 > 包级（`resolver.ts:51-55`） | 核心不可修改，领域特定需独立插件（`CLAUDE.md:53`） |

OpenSpec 通过三层 Schema 优先级实现"项目可定制、用户可覆盖、包内提供默认"的渐进式扩展。Superpowers 通过零依赖策略确保在所有编码代理中可运行，但将领域特定扩展排斥在核心之外。

### 维度四：结论——互补关系

两个项目的哲学并不矛盾，而是互补：

- **OpenSpec 治"怎么走"**：通过 DAG 声明任务间的数据流关系，让 AI 代理知道"下一步可以做什么"
- **Superpowers 治"别跑偏"**：通过 Red Flags 和合理化借口表，让 AI 代理知道"什么时候必须停下来"

用一句话总结：OpenSpec 提供地图，Superpowers 提供护栏。两者组合使用，AI 代理既知道该去哪里，也不会抄近路翻车。

## 五、clipforge 应用建议

将上述设计哲学映射到 clipforge 管线改造：

### 1. 采用 OpenSpec 的"依赖是使能者"哲学

**做法**：clipforge 的 `schema.yaml`（或等效的阶段配置）中，`requires` 字段只表示"输入文件已就绪"，不表示"必须严格按序执行"。例如 Stage 5（素材准备）与 Stage 4（音频生产）如果互相不依赖对方的输出，应允许并行触发。

**依据**：OpenSpec 的 "Dependencies are enablers, not gates"（`concepts.md:592`）

### 2. 采用 OpenSpec 的"状态即文件"

**做法**：每个 Stage 的完成状态 = `generates` 字段声明的文件是否存在。例如 Stage 4 完成的判定标准是 `workspace/audio/` 目录下存在 `narration.mp3`（或分段 TTS 文件）和 `bgm.mp3`。无需额外的状态数据库或标志位。

**依据**：OpenSpec 的 Schema 中每个 artifact 通过 `generates` 字段声明输出文件（`concepts.md:554-569`），resolver 通过 `fs.existsSync` 检查文件存在性（`resolver.ts:70-71`）

### 3. 采用 Superpowers 的 Anti-rationalization 模式

**做法**：为每个 Stage 技能增加 Red Flags 表。例如 Stage 6（视频制作）的 Red Flags 应包含：
- "跳过了封面检查就继续" → 必须验证封面帧存在
- "视频时长与预期不符就进入交付" → 必须重新生成
- "旁白音画不同步就算完成" → 必须校准时间轴

**依据**：Superpowers 的 Red Flags 列表模式（`SKILL.md:513-522`）和合理化借口表（`SKILL.md:446-457`）

### 4. 采用 Superpowers 的 CSO 原则

**做法**：clipforge 各 Stage 技能的 `description` 字段只写触发条件，不写流程摘要。例如 Stage 7 的描述应为 "Use when Stage 6 video file exists and needs cover image, platform copy, and delivery packaging"，而非 "生成封面图、撰写抖音文案、打包交付物"。

**依据**：CSO 原则明确禁止在描述中总结工作流（`SKILL.md:150-153`），实测证明工作流摘要会导致代理跳过技能正文（`SKILL.md:155-158`）

### 5. 采用 Superpowers 的 Prompt 模板化

**做法**：将 SubAgent 的执行 prompt 与技能内容分离。技能 SKILL.md 只定义"做什么"和"不做什么"，具体的 SubAgent prompt 模板作为独立文件存储（类似 Superpowers 的 `subagent-driven-development` 将任务描述与技能流程分离的做法）。这样修改 prompt 不影响技能逻辑，修改技能逻辑不需要重写 prompt。

**依据**：Superpowers 的 subagent-driven-development 模式（`README.md:162`）将任务执行与审查分离，技能文档中的流程图就是子代理的执行模板
