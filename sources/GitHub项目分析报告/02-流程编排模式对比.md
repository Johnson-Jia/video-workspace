# 流程编排模式对比：OpenSpec vs Superpowers

## 一、核心概念提炼

### 1. DAG 依赖图编排（OpenSpec）

OpenSpec 将工作流建模为有向无环图（DAG）。每个 artifact 在 YAML 中声明自己的 `requires` 依赖列表，系统通过 Kahn 拓扑排序算法自动推导合法的执行顺序。运行时通过 `getNextArtifacts()` 动态计算当前哪些 artifact 的依赖已全部满足、可以被构建，无需在代码或 prompt 中硬编码执行序列。只要修改 YAML 定义即可改变流程拓扑，编排逻辑完全由数据驱动。

> 源文件：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\graph.ts`（`getBuildOrder()` 方法，第 72-113 行）、`D:\AI-Agent\github-analyze\OpenSpec\schemas\spec-driven\schema.yaml`（artifacts 定义及 requires 字段）

### 2. 显式 SUB-SKILL 引用链编排（Superpowers）

Superpowers 通过技能文件末尾的显式引用构成执行链。每个 SKILL.md 在流程末尾声明下一步要调用的技能名称（如 brainstorming 的第 9 项 "Transition to implementation -- invoke writing-plans"），形成 brainstorming -> writing-plans -> subagent-driven-development -> finishing-a-development-branch 的固定引用链。每个技能内部用编号 checklist 定义步骤，子代理通过 prompt 模板注入上下文，三角色循环（Implementer -> Spec Reviewer -> Code Quality Reviewer）在任务级别形成内层编排。

> 源文件：`D:\AI-Agent\github-analyze\superpowers\skills\brainstorming\SKILL.md`（Checklist 第 9 项，第 32 行）、`D:\AI-Agent\github-analyze\superpowers\skills\writing-plans\SKILL.md`（Plan Document Header "REQUIRED SUB-SKILL"，第 53 行）、`D:\AI-Agent\github-analyze\superpowers\skills\subagent-driven-development\SKILL.md`（三角色循环流程图，第 46-86 行）

### 3. 硬编码 stage 顺序编排（当前 clipforge）

clipforge 在编排文件（如 `github-daily-trending.md`）中硬编码了 Stage 0 到 Stage 7 的固定执行顺序，每个 SubAgent 的 prompt 中直接写明要读取哪个阶段文件、产出什么文件。阶段间的状态传递通过一个中心化的 `stage-handoff.json` 文件实现，SubAgent 完成后由编排者通过 `ls` 命令验证产出文件是否存在。并行关系（如 Stage 4 与 Stage 5）在编排文件的文字描述中声明，但实际由编排者手动分派。

> 源文件：`D:\AI-Agent\video-clipforge\.claude\commands\github-daily-trending.md`（Step 5 中 SubAgent-1 到 SubAgent-5 的硬编码分派，第 170-351 行）、`D:\AI-Agent\video-clipforge\.claude\commands\clipforge.md`（工作流表格，第 63-74 行）

## 二、OpenSpec 做法

### 2.1 schema.yaml 声明 artifact 及其 requires 依赖

在 YAML 定义文件中，每个 artifact 由 `id`、`generates`（输出路径）、`description`、`template`、`instruction`、`requires` 字段组成。`requires` 是一个字符串数组，列出当前 artifact 依赖的其他 artifact ID。例如 `spec-driven` 内置 schema 中，`specs` 依赖 `proposal`，`tasks` 同时依赖 `specs` 和 `design`，`design` 依赖 `proposal`，形成 DAG 拓扑。apply 阶段通过 `apply.requires` 声明需要哪些 artifact 完成后才能进入实施。

> 文件：`D:\AI-Agent\github-analyze\OpenSpec\schemas\spec-driven\schema.yaml`（第 1-154 行，artifacts 数组及 apply 字段）

### 2.2 ArtifactGraph 用 Kahn 算法做拓扑排序

`ArtifactGraph` 类的核心方法 `getBuildOrder()` 实现了 Kahn 拓扑排序算法。它先计算每个 artifact 的入度（即 `requires.length`），构建反向邻接表（谁依赖谁），然后将入度为 0 的 artifact 入队。每次出队一个 artifact，将其所有下游依赖的入度减 1，入度降为 0 的继续入队。最终返回一个合法的构建序列。排序过程中对每批新就绪的 artifact 做 `sort()` 保证确定性。

> 文件：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\graph.ts`（`getBuildOrder()` 方法，第 72-113 行）

### 2.3 detectCompleted 通过检查文件系统判断完成状态

`detectCompleted()` 函数遍历 DAG 中的所有 artifact，对每个 artifact 调用 `isArtifactComplete()`。后者通过 `artifactOutputExists()` 检查 artifact 的 `generates` 路径（支持普通路径和 glob 模式）对应的文件是否存在于磁盘。如果文件存在，该 artifact 被标记为已完成，加入 `CompletedSet`（一个 `Set<string>`）。这意味着 OpenSpec 不依赖任何显式的状态标记文件，而是直接从文件系统推断进度。

> 文件：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\state.ts`（`detectCompleted()` 函数，第 14-29 行）
>
> 辅助文件：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\outputs.ts`（`artifactOutputExists()` 函数，第 40-42 行，通过 `resolveArtifactOutputs()` 解析路径并检查文件存在性）

### 2.4 generateInstructions 组装 AI 指令

`generateInstructions()` 函数将多个信息源组装成结构化的 `ArtifactInstructions` 对象，供 AI 消费。组装顺序为：先加载模板内容（`loadTemplate()`），再获取依赖信息（`getDependencyInfo()`，标注每个依赖是否完成、输出路径、描述），然后获取当前 artifact 完成后会解锁哪些下游 artifact（`getUnlockedArtifacts()`）。此外还从项目配置中提取 `context`（项目级背景信息）和 `rules`（artifact 级约束规则），分别作为独立字段传入，不拼接到模板中。

> 文件：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\instruction-loader.ts`（`generateInstructions()` 函数，第 275-342 行，以及注释中的 "Instruction injection order"，第 276-279 行）

### 2.5 AI 通过 CLI 命令获取上下文

AI 在执行时通过调用 `openspec status --json` 获取当前所有 artifact 的状态（done / ready / blocked），通过 `openspec instructions --json` 获取下一个待执行 artifact 的完整指令（模板 + 上下文 + 规则 + 依赖信息 + 解锁预览）。`formatChangeStatus()` 函数生成 `ChangeStatus` 结构，包含 `nextSteps`（人类可读建议）和 `actionContext`（机器可读约束），两者共同引导 AI 的下一步行为。

> 文件：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\instruction-loader.ts`（`formatChangeStatus()` 函数，第 488-548 行，以及 `ChangeStatus` 接口定义，第 133-144 行）
>
> 类型定义：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\types.ts`（Zod schema 定义，`ArtifactSchema`、`SchemaYamlSchema`，第 4-31 行）

## 三、Superpowers 做法

### 3.1 技能末尾显式引用下一个技能

### 3.2 SubAgent 三角色循环

subagent-driven-development 为每个任务分派三个不同角色的 SubAgent，形成两阶段审查循环。首先分派 **Implementer SubAgent**（使用 `implementer-prompt.md` 模板）执行实现、测试、提交和自查。Implementer 完成后，分派 **Spec Compliance Reviewer SubAgent**（使用 `spec-reviewer-prompt.md` 模板）验证实现是否与规格完全匹配（不多不少）。Spec 审查通过后，再分派 **Code Quality Reviewer SubAgent**（使用 `code-quality-reviewer-prompt.md` 模板）检查代码质量。审查发现问题则回到 Implementer 修复，修复后重新审查，直到通过。

Superpowers 强制要求审查顺序：必须先通过 spec compliance review，再进行 code quality review。在 Red Flags 部分明确写道："Start code quality review before spec compliance is [checked] (wrong order)" 和 "Move to next task while either review has open issues"。Spec 审查的核心原则是"Do Not Trust the Report"——不信任 Implementer 的自报结果，必须独立阅读代码与规格逐行比对。Code quality 审查在 spec 审查通过后才启动，关注代码组织、测试质量、可维护性。

> 文件：`D:\AI-Agent\github-analyze\superpowers\skills\subagent-driven-development\code-quality-reviewer-prompt.md`（"Only dispatch after spec compliance review passes"，第 7 行）

### 3.4 每个 SubAgent 角色使用不同级别的模型

Model Selection 部分根据任务复杂度选择不同能力的模型。**机械实现任务**（1-2 个文件，规格清晰）使用最快最便宜的模型；**集成和判断任务**（多文件协调、模式匹配、调试）使用标准模型；**架构、设计和审查任务**使用最强模型。任务复杂度的判断信号包括：涉及文件数、是否需要跨文件协调、是否需要设计判断或广泛理解代码库。Controller 在分派时根据任务特征选择合适的模型级别。

## 四、对比分析

| 维度 | OpenSpec | Superpowers | 当前 clipforge |
|------|----------|-------------|---------------|
| **编排方式** | DAG 数据结构驱动。`schema.yaml` 声明 artifact 及 `requires` 依赖，`ArtifactGraph` 类自动推导执行顺序 | prompt 链驱动。每个 SKILL.md 末尾显式引用下一个技能，形成固定引用链 | 硬编码顺序。编排文件中直接写死 SubAgent-1 到 SubAgent-5 的执行序列和阶段分配 |
| **并行支持** | DAG 自动推导。`getNextArtifacts()` 返回所有依赖已满足的 artifact，天然支持并行执行无依赖关系的 artifact | 手动 dispatch。writing-plans 的 Execution Handoff 提供两个选项（subagent-driven / inline），并行需人工判断 | 硬编码并行。编排文件文字描述中声明 Stage 4 与 Stage 5 可并行，但实际由编排者手动分派 |
| **状态检测** | 文件系统检测（代码实现）。`detectCompleted()` 通过检查 `generates` 路径对应的文件是否存在判断完成状态，支持 glob 模式 | 无显式状态检测。依赖 AI 按 checklist 逐项执行并汇报状态，审查通过后标记完成 | `stage-handoff.json` 中心化状态文件。编排者通过 `ls` 命令验证产出文件存在性 |
| **灵活性** | 高。修改 `schema.yaml` 即可改变流程拓扑（增删 artifact、调整依赖关系），无需改动代码 | 中。改变流程需修改 SKILL.md 中的引用链和 checklist，每个技能文件独立维护 | 低。改变流程需改编排文件中的 SubAgent prompt 和阶段分配，牵一发动全身 |
| **工具依赖** | 需要 CLI。AI 通过 `openspec status --json` 和 `openspec instructions --json` 获取上下文 | 无。纯 prompt 驱动，依赖 Skill 工具调用链 | 无。直接读文件 + Agent 工具分派 SubAgent |
| **审查机制** | 无内建审查。DAG 只负责依赖推导和状态检测，不涉及产出质量把关 | 两阶段审查。Spec Reviewer + Code Quality Reviewer 强制顺序执行，发现问题必须修复后重审 | 无内建审查。SubAgent 完成后仅做文件存在性验证，不做产出质量审查 |
| **上下文管理** | 结构化注入。`generateInstructions()` 将 template + context + rules + deps + unlocks 组装为 `ArtifactInstructions` 对象 | Controller 手动组装。编排者从 plan 中提取全部任务文本，拼接到 SubAgent prompt 中 | 编排文件硬编码。每个 SubAgent prompt 中直接写明要读取哪些文件、产出什么文件 |
| **错误恢复** | 重跑 artifact。重新运行 CLI 即可重新生成失败 artifact，已完成的 artifact 自动跳过 | 阶段内循环。Implementer 修复 -> Reviewer 重审，直到通过；无法解决则升级给人类 | 回退到指定阶段。通过错误恢复表手动选择回退路径，可能级联多个阶段 |

### 结论

**OpenSpec 的 DAG 适合复杂依赖推导。** 它的核心优势在于声明式依赖定义和自动拓扑排序，能够精确表达 artifact 之间的依赖关系（如 tasks 同时依赖 specs 和 design），运行时自动推导可并行批次。但 DAG 只解决"做什么、按什么顺序做"的问题，不解决"做得好不好"的问题。

**Superpowers 的 SubAgent 编排适合质量把控。** 它的核心优势在于三角色循环审查机制：先验证规格合规性（不多不少），再验证代码质量（干净可维护），每个角色有专门的 prompt 模板和行为约束。但它的编排是线性的技能引用链，无法表达并行或条件分支。

**clipforge 应结合两者。** 当前 clipforge 的硬编码编排方式在灵活性上最弱——改变流程需要大量重写编排文件中的 SubAgent prompt。但它在实践中有两个 Superpowers 不具备的特点：（1）通过 SubAgent 隔离解决上下文窗口问题（每个阶段独立上下文）；（2）通过 `stage-handoff.json` 和文件产出约定实现了跨阶段状态传递。这两个特点应保留并增强。

## 五、clipforge 应用建议

### 方案 C：混合 DAG + SubAgent 编排

以下 6 项改进建议，结合 OpenSpec 的声明式依赖管理优势和 Superpowers 的审查循环优势：

#### 1. 采用 OpenSpec 的 schema.yaml 格式定义 DAG，但不引入 CLI 依赖

使用与 OpenSpec 兼容的 YAML 格式定义 clipforge 的 stage 拓扑。每个 stage 声明 `id`、`generates`（产出文件路径）、`requires`（前置 stage ID）、`instruction`（阶段指令）、`template`（模板文件引用）。例如标准模式可以是：

```yaml
artifacts:
  - id: env_check
    generates: ".env-checked"
    requires: []
  - id: content
    generates: "raw_trending.json"
    requires: [env_check]
  - id: design
    generates: "design.md"
    requires: [content]
  - id: scenes
    generates: "narration_segments.json"
    requires: [design]
  - id: audio        # 与 assets 可并行
    generates: "segment_durations.json"
    requires: [scenes]
  - id: assets       # 可选，与 audio 可并行
    generates: "assets/manifest.md"
    requires: [scenes]
  - id: production
    generates: "output.mp4"
    requires: [audio]  # 不依赖 assets（可选素材）
  - id: delivery
    generates: "final.mp4"
    requires: [production]
```

**不引入 CLI 依赖的理由：** OpenSpec 的 CLI 工具需要 Node.js 运行时和 npm 包安装，对于 clipforge 的轻量级场景是过度依赖。改为 AI 直接解析 YAML 内容，在内存中构建等价的 DAG 逻辑（Kahn 算法本身不到 40 行代码）。

> 参考：`D:\AI-Agent\github-analyze\OpenSpec\schemas\spec-driven\schema.yaml`（artifact 数据结构）；`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\types.ts`（`ArtifactSchema` Zod 定义，第 4-11 行）；`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\graph.ts`（`getBuildOrder()` Kahn 算法，第 72-113 行）

#### 2. AI 解析 schema.yaml 后动态推导执行队列和 SubAgent 批次

AI 读取 YAML 定义后，在内存中执行等价于 `ArtifactGraph.getBuildOrder()` 的拓扑排序，然后调用等价于 `getNextArtifacts()` 的方法动态计算当前批次。每个批次内的 stage 无依赖关系，可并行分派 SubAgent。例如标准模式下，audio 和 assets 在同一批次（都只依赖 scenes），可并行执行；production 必须等 audio 完成后才能进入下一批次。

这种方式的好处是：修改流程只需改 YAML 文件（如新增一个"字幕生成"阶段、调整依赖关系），无需改动编排文件中的 SubAgent prompt。编排文件只需一个通用的"执行当前批次"逻辑。

> 参考：`D:\AI-Agent\github-analyze\OpenSpec\src\core\artifact-graph\graph.ts`（`getNextArtifacts()` 方法，第 118-134 行，动态计算所有依赖已满足的 artifact）；当前 clipforge 的硬编码分派（`D:\AI-Agent\video-clipforge\.claude\commands\github-daily-trending.md`，Step 5 SubAgent-1 到 SubAgent-5，第 170-351 行）

#### 3. 采用 Superpowers 的 SubAgent prompt 模板化

将当前编排文件中散落在各个 SubAgent prompt 中的重复内容（"你是 ClipForge 短视频制作流程的阶段执行者"、"技能库符号链接"等）抽取为参数化模板。参考 Superpowers 的 `implementer-prompt.md` 结构：Task Description（从 DAG 的 `instruction` 字段注入）-> Context（场景设定、依赖关系）-> Before You Begin（提问确认）-> Your Job（执行步骤）-> Report Format（产出汇报）。

每个模板文件对应一个角色：
- `prompts/implementer-prompt.md` — 阶段执行者模板（对应 Superpowers 的 Implementer）
- `prompts/spec-reviewer-prompt.md` — 产出审查者模板（对应 Superpowers 的 Spec Reviewer）

模板通过占位符（如 `{{ARTIFACT_ID}}`、`{{INSTRUCTION}}`、`{{DEPENDS_ON}}`）由编排者在分派时动态填充。

> 参考：`D:\AI-Agent\github-analyze\superpowers\skills\subagent-driven-development\implementer-prompt.md`（SubAgent prompt 模板结构，第 1-113 行，包含 Task Description / Context / Before You Begin / Your Job / Self-Review / Report Format 六段式结构）

#### 4. 采用 Superpowers 的三角色循环

在关键阶段（特别是 Stage 6 HTML 制作，历史上多次出现"只有背景没有内容"的事故）引入两阶段审查循环：

- **Stage 6 执行后：** 分派 Spec Reviewer SubAgent 验证 HTML 是否包含所有场景内容、`<audio>` 元素是否正确嵌入、GSAP timeline 是否注册、`window.__timelines["main"]` 是否存在。
- **Spec 审查通过后：** 分派 Code Quality Reviewer SubAgent 检查 CSS 是否使用 `.anim-in`（禁止）、HTML 是否包含实体字符（禁止）、`.scene-wrap` 是否有 padding。

审查未通过则回到 Implementer 修复，修复后重新审查。非关键阶段（如 Stage 0 环境检测）可跳过审查。
