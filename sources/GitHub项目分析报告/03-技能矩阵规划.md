# 技能矩阵规划：OpenSpec vs Superpowers → ClipForge 改造蓝图

## 一、核心概念提炼

通过对比 OpenSpec 和 Superpowers 两套技能体系，提炼出 AI 技能架构的 4 个关键维度：

### 1. 技能的定义单元

**什么构成一个"技能"？**

两个体系对"技能"的定义截然不同：

- **OpenSpec** 将技能定义为"一段可执行的指令文本"，由 TypeScript 函数生成。每个技能由 `SkillTemplate` 接口描述，包含 `name`、`description`、`instructions`、`license`、`compatibility`、`metadata` 六个字段。技能的本质是一个**自包含的 prompt 文件**，AI 读取后按步骤执行。
  > 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\templates\types.ts`，`SkillTemplate` 接口定义

- **Superpowers** 将技能定义为"经过 TDD 验证的行为文档"，以 Markdown 文件（SKILL.md）为载体。技能的核心不是步骤而是**行为约束**：Red Flags 列表、Anti-rationalization 表、Iron Law 声明。技能是对 AI 行为的"纪律塑造"而非"流程脚本"。
  > 来源：`D:\AI-Agent\github-analyze\superpowers\skills\writing-skills\SKILL.md`，"Writing skills IS Test-Driven Development applied to process documentation"

### 2. 技能的分类分层

**如何组织技能之间的关系？**

- **OpenSpec** 采用**工作流模板分类**：11 个工作流模板覆盖变更的完整生命周期（提议 → 探索 → 创建 → 继续 → 快进 → 应用 → 验证 → 同步 → 归档 → 批量归档 → 入门），每个模板生成一对 Skill + Command 文件。模板之间通过 `openspec` CLI 命令串联，形成状态机。
  > 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\templates\workflows\` 目录下 12 个 .ts 文件（含 feedback）

- **Superpowers** 采用**四层架构**：Layer 0 引导层（using-superpowers）→ Layer 1 流程层（brainstorming → writing-plans → subagent-driven）→ Layer 2 纪律层（TDD、debugging、verification）→ Layer 3 协作层（code-review 收发）→ Layer 4 元技能（writing-skills）。层级之间有明确的依赖关系和调用顺序。
  > 来源：`D:\AI-Agent\github-analyze\superpowers\skills\` 目录下 14 个技能目录

### 3. 技能的格式规范

**技能文件应该怎么写？**

- **OpenSpec** 的格式规范因目标工具而异：通过 `ToolCommandAdapter` 接口，同一份指令内容被格式化为不同工具的 frontmatter 格式。Claude Code 用 YAML frontmatter（name/description/category/tags），Cursor 用不同的 frontmatter 字段，Codex 用绝对路径。技能内容本身是纯文本指令。
  > 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\command-generation\types.ts`，`ToolCommandAdapter` 接口

- **Superpowers** 有统一的 SKILL.md 格式规范：YAML frontmatter（仅 name + description，总计 1024 字符以内）+ 结构化正文（Overview / When to Use / Core Pattern / Quick Reference / Implementation / Common Mistakes）。强调 CSO 原则（Claude Search Optimization）：description 只写触发条件，不写工作流程。
  > 来源：`D:\AI-Agent\github-analyze\superpowers\skills\writing-skills\SKILL.md`，"CSO" 和 "SKILL.md Structure" 章节

### 4. 技能的发现/触发机制

**AI 如何找到并激活技能？**

- **OpenSpec** 采用**CLI 注册 + 自动发现**：通过 `openspec` CLI 命令将技能文件安装到目标工具的 commands 目录，AI 通过斜杠命令（如 `/opsx:propose`）触发。技能文件放在约定路径（如 `.claude/commands/opsx/explore.md`），AI 在启动时自动扫描可用命令。
  > 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\command-generation\adapters\claude.ts`，`getFilePath()` 返回 `.claude/commands/opsx/<id>.md`

- **Superpowers** 采用**Hook 注入 + 1% 规则**：`using-superpowers` 技能通过 SessionStart hook 注入到每次对话开始时，建立"1% 规则"——只要有 1% 的可能性某个技能适用，就必须调用它。AI 主动判断是否需要加载技能，通过 Skill 工具按需调用。
  > 来源：`D:\AI-Agent\github-analyze\superpowers\skills\using-superpowers\SKILL.md`，"The Rule" 和 "Red Flags" 章节

## 二、OpenSpec 做法

### 2.1 工作流模板体系

OpenSpec 在 `src/core/templates/workflows/` 目录下定义了 12 个工作流模板（含 feedback），每个模板是一个 TypeScript 函数，返回 `SkillTemplate` 和可选的 `CommandTemplate` 对象。

| # | 工作流 | 文件 | 核心功能 |
|---|--------|------|---------|
| 1 | propose | `propose.ts` | 一步完成变更创建 + 全部 artifact 生成（proposal.md / design.md / tasks.md） |
| 2 | explore | `explore.ts` | 进入探索模式，作为"思考伙伴"而非执行者，禁止实现代码 |
| 3 | new-change | `new-change.ts` | 创建新变更目录，展示第一个 artifact 模板后停止等待用户指令 |
| 4 | continue-change | `continue-change.ts` | 继续未完成的变更，按依赖顺序创建下一个 artifact |
| 5 | ff-change | `ff-change.ts` | 快进模式：一次性创建所有实现所需的 artifact，不逐步确认 |
| 6 | apply-change | `apply-change.ts` | 实现 tasks.md 中的任务，逐个完成并标记 |
| 7 | verify-change | `verify-change.ts` | 验证实现是否匹配变更 artifact（规格/任务/设计三维检查） |
| 8 | sync-specs | `sync-specs.ts` | 将 delta specs 同步到主 specs，支持 ADDED/MODIFIED/REMOVED/RENAMED |
| 9 | archive-change | `archive-change.ts` | 归档已完成的变更，检查 artifact 和 task 完成度 |
| 10 | bulk-archive-change | `bulk-archive-change.ts` | 批量归档多个并行变更，处理 spec 冲突 |
| 11 | onboard | `onboard.ts` | 引导式入门，带用户走完一个完整的变更周期 |
| 12 | feedback | `feedback.ts` | 收集并提交用户反馈，含匿名化处理 |

> 文件路径：`D:\AI-Agent\github-analyze\OpenSpec\src\core\templates\workflows\` 目录下各 .ts 文件

每个模板的核心结构相同：导出一个 `getXxxSkillTemplate()` 函数，返回包含 `name`、`description`、`instructions` 的对象。部分模板额外导出 `getXxxCommandTemplate()` 生成斜杠命令版本。instructions 是完整的 Markdown 指令文本，包含 Steps、Output、Guardrails 三个标准段落。

### 2.2 适配器体系

OpenSpec 在 `src/core/command-generation/adapters/` 目录下定义了 27 个工具适配器，全部实现统一的 `ToolCommandAdapter` 接口。

**`ToolCommandAdapter` 接口定义**（`types.ts`）：

```typescript
export interface ToolCommandAdapter {
  toolId: string;                                    // 工具标识，如 'claude', 'cursor'
  getFilePath(commandId: string): string;            // 返回文件路径，如 '.claude/commands/opsx/explore.md'
  formatFile(content: CommandContent): string;       // 格式化完整文件内容（含 frontmatter）
}
```

> 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\command-generation\types.ts`，第 32-48 行

**27 个适配器清单**（按 `index.ts` 导出顺序）：

amazon-q、antigravity、auggie、bob、claude、cline、codex、codebuddy、continue、costrict、crush、cursor、factory、gemini、github-copilot、iflow、junie、kilocode、kiro、opencode、pi、qoder、lingma、qwen、roocode、windsurf

> 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\command-generation\adapters\index.ts`

**适配器差异化示例**：

- **claude.ts**：路径 `.claude/commands/opsx/<id>.md`，frontmatter 含 name/description/category/tags 四个字段，YAML 值经过转义处理
- **factory.ts**：路径 `.factory/commands/opsx-<id>.md`，frontmatter 仅 description + argument-hint 两个字段
- **codex.ts**：使用绝对路径（全局作用域），与其他工具的项目级路径不同

> 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\command-generation\adapters\claude.ts` 和 `factory.ts`

### 2.3 类型体系

**`SkillTemplate` 接口**（`templates/types.ts`）：

```typescript
export interface SkillTemplate {
  name: string;
  description: string;
  instructions: string;
  license?: string;
  compatibility?: string;
  metadata?: Record<string, string>;
}
```

**`CommandTemplate` 接口**：

```typescript
export interface CommandTemplate {
  name: string;
  description: string;
  category: string;
  tags: string[];
  content: string;
}
```

> 来源：`D:\AI-Agent\github-analyze\OpenSpec\src\core\templates\types.ts`

### 2.4 总结

OpenSpec 的技能体系核心特征：

1. **工作流模板是 TypeScript 函数，生成 Skill 文件**——技能是代码的输出，而非代码本身。开发者修改 TypeScript 源码，运行 CLI 后重新生成各工具的技能文件。
2. **27 个适配器实现统一 ToolCommandAdapter 接口**——同一份指令内容通过不同适配器格式化为 Claude Code / Cursor / Codex 等 27 种工具的本地格式，实现"写一次、到处部署"。
3. **三层 Schema 解析**——项目级 `.openspec.yaml` > 用户级全局配置 > 包级默认值，Schema 定义变更的 artifact 依赖图和生命周期。
4. **技能通过 CLI 命令注册，AI 通过斜杠命令发现和激活**——技能文件放在工具约定的目录下，AI 启动时自动扫描可用命令列表。

### 3.2 技能编写规范（writing-skills）

这是 Superpowers 的元技能，定义了技能本身应该如何编写。

**CSO 原则（Claude Search Optimization）**：

description 字段只写触发条件（"Use when..."），绝不写技能的工作流程或摘要。原因：测试发现，当 description 摘要了技能流程时，AI 会直接跟随 description 而跳过读取完整技能内容。例如 description 写 "code review between tasks" 导致 AI 只做一次 review，而技能正文实际要求两次（规格合规 + 代码质量）。

> 来源：`D:\AI-Agent\github-analyze\superpowers\skills\writing-skills\SKILL.md`，"Claude Search Optimization (CSO)" 章节，第 141-172 行

**格式约束**：

1. YAML frontmatter 仅有两个必填字段：`name`（字母+数字+连字符）和 `description`（500 字符以内）
2. 正文结构：Overview → When to Use → Core Pattern → Quick Reference → Implementation → Common Mistakes
3. 辅助文件分离规则：100+ 行的重型参考独立文件，prompt 模板独立文件，原则和短代码片段保持内联

> 来源：`D:\AI-Agent\github-analyze\superpowers\skills\writing-skills\SKILL.md`，"SKILL.md Structure" 和 "File Organization" 章节

### 3.3 技能发现和触发机制（using-superpowers）

**1% 规则**：

```
If you think there is even a 1% chance a skill might apply to what you are doing,
you ABSOLUTELY MUST invoke the skill.
```

这个规则通过 SessionStart hook 注入到每次对话开始。AI 在收到任何用户消息后，必须先检查是否有适用的技能，即使只有 1% 的可能性也要调用。调用后发现不适用可以跳过，但不调用是违规行为。

**Red Flags 列表**（防止 rationalization）：

| Thought | Reality |
|---------|---------|
| "This is just a simple question" | Questions are tasks. Check for skills. |
| "I need more context first" | Skill check comes BEFORE clarifying questions. |
| "Let me explore the codebase first" | Skills tell you HOW to explore. Check first. |
| "This doesn't need a formal skill" | If a skill exists, use it. |

> 来源：`D:\AI-Agent\github-analyze\superpowers\skills\using-superpowers\SKILL.md`，"The Rule" 和 "Red Flags" 章节

### 3.4 流程技能示例（brainstorming）

brainstorming 技能展示了 Superpowers 流程层的设计模式：

- **9 步 Checklist**：从探索项目上下文到过渡到 writing-plans，每步都有明确的输入/输出
- **HARD-GATE**："Do NOT invoke any implementation skill, write any code... until you have presented a design and the user has approved it"
- **Anti-Pattern 显式声明**："This Is Too Simple To Need A Design" 被直接列为反模式

### 3.5 纪律技能示例（test-driven-development）

TDD 技能展示了 Superpowers 纪律层的设计模式：

- **Iron Law**："NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"，大写加粗，无例外

- **Red Flags 列表**：13 条红旗信号，以 "All of these mean: Delete code. Start over with TDD." 结尾

## 四、对比分析

### 4.1 全维度对比

| 维度 | OpenSpec | Superpowers |
|------|----------|-------------|
| **技能生成方式** | 代码生成（TypeScript 函数 → SkillTemplate） | 手动编写（Markdown SKILL.md） |
| **分类体系** | 工作流模板（12 个）+ 适配器（27 个） | 四层架构（14 个技能） |
| **格式规范** | 各工具不同 frontmatter（通过适配器格式化） | 统一 SKILL.md 格式（name + description + 结构化正文） |
| **发现机制** | CLI 注册到约定目录 + 斜杠命令触发 | Hook 注入 + 1% 规则 + Skill 工具按需调用 |
| **扩展方式** | 添加适配器（实现 ToolCommandAdapter 接口） | 添加技能目录（含 SKILL.md + 可选辅助文件） |
| **纪律机制** | 无显式纪律约束，靠 Guardrails 段落引导 | Red Flags + Anti-rationalization Table + Iron Law |
| **核心关注点** | 工具适配（27 种 AI 工具格式兼容） | 行为塑造（AI 遵循纪律、不找借口） |
| **状态管理** | CLI 管理变更状态（`openspec status --json`） | 无显式状态管理，靠技能间引用串联 |
| **Artifact 定义** | Schema YAML 定义 artifact 依赖图 | 无显式 artifact 概念，技能间通过内容引用 |
| **测试要求** | 无 | 技能必须 TDD 验证（RED-GREEN-REFACTOR） |

### 4.2 设计哲学差异

**OpenSpec 侧重工具适配**：核心价值是"写一次指令，部署到 27 种工具"。技能是可生成的、格式化的、通过 CLI 管理的产物。适用场景：跨工具团队协作，需要统一的变更管理流程。

**Superpowers 侧重行为塑造**：核心价值是"让 AI 遵循纪律，抵抗合理化借口"。技能是经过 TDD 验证的、包含 Anti-rationalization 的行为约束文档。适用场景：AI 辅助开发中的质量保证，防止 AI 走捷径。

### 4.3 互补关系

两个体系不冲突，可组合使用：

- OpenSpec 提供"变更管理基础设施"——Schema 定义、Artifact DAG、CLI 状态管理、多工具格式适配
- Superpowers 提供"行为纪律层"——1% 规则确保技能被使用，Red Flags 防止跳过关键步骤，Anti-rationalization 封堵借口

ClipForge 的 tbcflow 编排正是这种组合的实践：OpenSpec 的 propose/apply 工作流作为变更管理骨架，Superpowers 的 brainstorming/writing-plans/subagent-driven 作为行为纪律约束。
