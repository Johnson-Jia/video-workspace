第一章：项目概览

OpenSpec 是什么？

OpenSpec 是一个 AI 原生的规范驱动开发框架，帮助你和 AI 编码助手在写代码之前就对齐要做什么。它在你现有的项目中创建一个轻量级的规范层，让 AI 的输出更可预测、更可控。

一句话理解

OpenSpec = 规范先行 + AI 编码。先写清楚"要做什么"，再让 AI 动手。

核心理念

流动而非僵化

没有阶段门槛，随时可以做任何操作。

迭代而非瀑布

边构建边学习，随时回溯优化。

简单而非复杂

秒级初始化，最小仪式感。

面向棕地

专为已有项目设计，不只适用于全新项目。

它怎么运作的？

OpenSpec 在你的项目中创建 openspec/ 目录，包含两个核心区域：

specs/ — 系统真相

描述系统当前行为的规范。按领域组织（如 auth/、payments/），包含需求和场景。

changes/ — 变更提案

每个变更一个文件夹，包含提案、规范差异、设计、任务。完成后归档并合并到主规范。

变更完成归档后，差异规范会自动合并到主规范中：

openspec/
├── specs/              # 系统真相（当前行为规范）
│   └── auth/
│       └── spec.md
├── changes/            # 变更提案（一个文件夹 = 一个变更）
│   └── add-dark-mode/
│       ├── proposal.md  # 为什么做 + 做什么
│       ├── specs/       # 规范差异（新增/修改/删除）
│       ├── design.md    # 技术方案
│       └── tasks.md     # 实施清单
└── config.yaml         # 项目配置（可选）

第二章：为什么需要 OpenSpec

AI 编码助手很强大，但当需求只存在于聊天历史中时，输出往往不可预测。OpenSpec 添加了一个轻量级规范层，让你在代码编写之前就和 AI 对齐。

先对齐再动手

人和 AI 在写代码前先在规范上达成一致。

有条理

每个变更都有独立的文件夹：提案、规范、设计、任务。

灵活工作

随时更新任何产物，没有僵化的阶段门槛。

工具无关

支持 25+ AI 编码工具，用你喜欢的就行。

第三章：与竞品对比

方案优点缺点适合谁
OpenSpec 轻量、灵活、支持 25+ 工具、可自定义工作流 相对年轻，社区还在成长 大多数开发者
GitHub Spec Kit 全面、由 GitHub 维护 重量级、阶段门槛多、需要 Python 大型企业

第四章：环境要求

版本要求

OpenSpec 需要 Node.js 20.19.0 或更高版本。请先确认你的 Node.js 版本。

4.1 检查 Node.js 版本

验证 Node.js
node --version

应输出 v20.19.0 或更高。如果版本过低或未安装，按以下方式升级。

4.2 安装/升级 Node.js（如果需要）

1推荐使用 fnm 管理 Node 版本
# macOS / Linux
curl -fsSL https://fnm.vercel.app/install | bash
source ~/.bashrc

# Windows（PowerShell 管理员模式）
winget install Schniz.fnm

# 然后安装 Node.js 24
fnm install 24
fnm use 24
fnm default 24

也可以直接从 nodejs.org 下载安装。

安装前检查清单

  • Node.js >= 20.19.0
  • 一个 AI 编码工具（Claude Code、Cursor、Windsurf 等，详见第七章）
  • 一个现有项目（OpenSpec 专为已有项目设计）
  • 终端 / 命令行工具

第五章：安装 OpenSpec CLI

OpenSpec 支持多种包管理器安装。选择你习惯的方式：

npm install -g @fission-ai/openspec@latest
pnpm add -g @fission-ai/openspec@latest
yarn global add @fission-ai/openspec@latest
bun add -g @fission-ai/openspec@latest

Bun 可以全局安装 OpenSpec，但 OpenSpec 当前运行在 Node.js 上，仍需 Node.js >= 20.19.0。

Nix 用户（免安装直接运行）

# 直接运行
nix run github:Fission-AI/OpenSpec -- init

# 或安装到 profile
nix profile install github:Fission-AI/OpenSpec

验证安装

确认安装成功
openspec --version

应输出 1.3.1 或更高版本号。

第六章：初始化项目

安装 CLI 后，在你的项目目录中初始化 OpenSpec：

1进入你的项目目录
cd your-project
2运行初始化
openspec init

初始化会交互式引导你选择：

  • AI 工具选择 — 你使用的编码助手（Claude Code、Cursor 等）
  • 工作流配置 — 默认使用 core 配置（propose / explore / apply / sync / archive）
  • 项目配置 — 可选的 config.yaml，输入技术栈和约定
3非交互式初始化（CI/脚本）
# 指定工具
openspec init --tools claude,cursor

# 全部工具
openspec init --tools all

# 跳过工具配置
openspec init --tools none

# 自动清理旧文件
openspec init --force

初始化创建了什么？

openspec/
├── specs/              # 你的规范（真相来源）
├── changes/            # 变更提案
└── config.yaml         # 项目配置

.claude/skills/         # Claude Code 技能文件（如选择了 Claude）
.cursor/skills/         # Cursor 技能文件（如选择了 Cursor）
...                     # 其他工具的配置

初始化后立刻可用

初始化完成后，你的 AI 编码助手会自动检测 OpenSpec 的技能文件。无需重启，直接在聊天中使用 /opsx:propose 即可开始。

第七章：第一次变更 — 完整实战

让我们用一个真实例子走完整个流程：为你的应用添加暗黑模式。

7.1 创建变更提案

1告诉 AI 你的想法

在你的 AI 编码助手聊天界面输入：

/opsx:propose add-dark-mode

AI 会自动创建变更目录并生成所有规划产物。

2查看 AI 生成了什么
AI:  Created openspec/changes/add-dark-mode/
     ✓ proposal.md — 为什么做、做什么
     ✓ specs/       — 需求和场景
     ✓ design.md    — 技术方案
     ✓ tasks.md     — 实施清单
     Ready for implementation!

7.2 产物说明

proposal.md

为什么做这件事 + 做什么。捕获意图、范围、高层方案。

specs/

差异规范，描述新增/修改/删除的需求和场景。

design.md

技术方案：架构决策、数据流、文件变更列表。

tasks.md

实施清单：带复选框的任务列表，按编号组织。

7.3 产物之间的依赖关系

proposal
→
specs + design
→
tasks
→
实施

依赖是"赋能"关系，不是"门控"关系 — 你可以随时跳过或回溯

7.4 实施任务

3让 AI 开始实施
/opsx:apply

AI 会读取 tasks.md，逐个完成任务并打勾。

AI:  Working through tasks...
     ✓ 1.1 Created ThemeContext with light/dark state
     ✓ 1.2 Added CSS custom properties to globals.css
     ✓ 1.3 Implemented localStorage persistence
     ✓ 2.1 Created ThemeToggle component
     ...
     All tasks complete!

7.5 归档变更

4归档完成的变更
/opsx:archive
AI:  Archiving add-dark-mode...
     ✓ Merged specs into openspec/specs/ui/spec.md
     ✓ Moved to openspec/changes/archive/2026-05-24-add-dark-mode/
     Done! Ready for the next feature.

恭喜！你已完成第一个 OpenSpec 工作循环

差异规范已合并到主规范中，变更已归档保留。你的 specs/ui/spec.md 现在包含了暗黑模式的行为描述。

第八章：核心工作流（默认 core 配置）

新安装默认使用 core 配置，包含 5 个命令：

/opsx:propose
→
/opsx:apply
→
/opsx:sync
→
/opsx:archive
命令用途说明
/opsx:propose创建变更 + 生成规划产物一步到位的默认起始命令
/opsx:explore探索想法需求不明确时先探索，不创建任何文件
/opsx:apply实施任务按 tasks.md 逐个完成，支持中断恢复
/opsx:sync同步差异规范可选。归档时会自动提示同步
/opsx:archive归档变更合并差异规范 + 移动到 archive/

第九章：扩展工作流

如果你需要更精细的控制，可以启用扩展工作流命令：

1启用扩展工作流
# 配置工作流配置
openspec config profile

# 刷新项目文件
openspec update

扩展命令一览

命令用途场景
/opsx:new创建变更脚手架只想创建空目录，逐步填充
/opsx:continue创建下一个产物一步一步来，每步审查
/opsx:ff快进：一次创建所有规划产物清楚要做什么时使用
/opsx:verify验证实施结果归档前检查实施是否匹配规范
/opsx:bulk-archive批量归档同时完成多个变更
/opsx:onboard交互式教程新用户学习完整流程

使用场景对比

什么时候用 /opsx:ff，什么时候用 /opsx:continue？

情况用哪个
需求清晰，准备好开工/opsx:ff
还在探索，想逐步审查/opsx:continue
时间紧，需要快速推进/opsx:ff
复杂变更，想精细控制/opsx:continue

第十章：探索模式

当你不确定要做什么，或者需要先调查问题时：

开启探索对话
/opsx:explore
You: /opsx:explore

AI:  What would you like to explore?

You: 我该用什么方案处理 API 限流？

AI:  让我看看你的现有设置...
     [分析代码库]

     我看到三种方案：
     1. 中间件方式（简单，按端点限制）
     2. Redis 后端（可扩展，共享状态）
     3. 每用户令牌桶（公平，但复杂）

     你的规模怎么样？

You: 中型项目，先从简单的开始吧。
AI:  那方案 1 最合适。准备好后运行 /opsx:propose add-rate-limiting 开始。

探索模式不会创建任何文件。纯粹是思考伙伴，帮你理清思路。

第十一章：项目配置

通过 openspec/config.yaml 定制 OpenSpec 的行为：

1创建配置文件
# openspec/config.yaml
schema: spec-driven

context: |
  技术栈: TypeScript, React, Node.js
  API 风格: RESTful, JSON 响应
  测试: Vitest 单元测试 + Playwright E2E
  代码风格: ESLint + Prettier, 严格 TypeScript

rules:
  proposal:
    - 包含回滚方案
    - 标注受影响的团队
  specs:
    - 使用 Given/When/Then 格式
  design:
    - 复杂流程包含时序图

配置字段说明

字段类型说明
schemastring默认工作流 Schema（如 spec-driven）
contextstring项目上下文，注入到所有产物（上限 50KB）
rulesobject按产物 ID 分组的规则（仅注入到对应产物）

Schema 解析优先级（从高到低）

  1. CLI 标志：--schema <name>
  2. 变更元数据：.openspec.yaml
  3. 项目配置：config.yaml
  4. 默认值：spec-driven

第十二章：多语言支持

在 config.yaml 的 context 中添加语言指令即可让所有产物使用指定语言。

简体中文配置

# openspec/config.yaml
schema: spec-driven

context: |
  语言：中文（简体）
  所有产出物必须用简体中文撰写。
  技术术语如 "API"、"REST"、"GraphQL" 保持英文。
  代码示例和文件路径保持英文。

  技术栈: TypeScript, React 18, Node.js 20

其他语言示例

语言context 设置
日语言語：日本語 / すべての成果物は日本語で作成してください
葡萄牙语Language: Portuguese (pt-BR) / All artifacts must be written in Brazilian Portuguese
西班牙语Idioma: Español / Todos los artefactos deben escribirse en español
法语Langue: Français / Tous les artefacts doivent être rédigés en français

第十三章：自定义 Schema

当项目配置不够灵活时，你可以创建完全自定义的工作流。

方式一：Fork 内置 Schema

1复制内置 Schema
openspec schema fork spec-driven my-workflow

这会将 spec-driven 完整复制到 openspec/schemas/my-workflow/，你可以自由编辑。

方式二：从零创建

1创建新 Schema
# 交互式
openspec schema init research-first

# 非交互式
openspec schema init rapid \
  --description "快速迭代工作流" \
  --artifacts "proposal,tasks" \
  --default

Schema 结构

openspec/schemas/my-workflow/
├── schema.yaml           # 工作流定义
└── templates/
    ├── proposal.md       # 提案模板
    ├── spec.md           # 规范模板
    ├── design.md         # 设计模板
    └── tasks.md          # 任务模板

schema.yaml 示例

name: my-workflow
version: 1
description: 我的团队自定义工作流

artifacts:
  - id: proposal
    generates: proposal.md
    description: 变更提案
    template: proposal.md
    instruction: |
      创建提案，解释为什么需要这个变更。
      聚焦问题本身，不要过早给出方案。
    requires: []

  - id: design
    generates: design.md
    description: 技术设计
    template: design.md
    instruction: |
      创建技术设计文档，解释如何实施。
    requires:
      - proposal

  - id: tasks
    generates: tasks.md
    description: 实施清单
    template: tasks.md
    requires:
      - design

apply:
  requires: [tasks]
  tracks: tasks.md
验证 Schema
openspec schema validate my-workflow

第十六章：协调工作区（Beta）

Beta 功能

工作区功能正在积极开发中，命令和输出格式可能随时变更。不建议在生产环境中依赖。

当一个项目跨越多个仓库或文件夹时，协调工作区提供统一的规划中心：

# 引导式创建工作区
openspec workspace setup

# 非交互式创建
openspec workspace setup --no-interactive \
  --name platform \
  --link /repos/api \
  --link web=/repos/web

# 查看已知工作区
openspec workspace list

# 检查工作区状态
openspec workspace doctor

# 打开工作区
openspec workspace open

工作区结构

workspace-folder/
├── changes/                       # 工作区级规划
└── .openspec-workspace/
    ├── workspace.yaml             # 共享身份和链接名
    └── local.yaml                 # 本机路径映射

第十七章：从旧版迁移

如果你之前使用旧版 OpenSpec（/openspec:proposal 等命令），迁移非常简单：

1升级 CLI
npm install -g @fission-ai/openspec@latest
2运行迁移
# 交互式
openspec init

# 或自动模式
openspec init --force

迁移映射

旧版命令新版等价
/openspec:proposal/opsx:propose（默认）或 /opsx:new + /opsx:ff
/openspec:apply/opsx:apply
/openspec:archive/opsx:archive

你的数据是安全的

  • 活跃变更 — 完整保留，可用新命令继续
  • 已归档变更 — 不受影响
  • 主规范 — 不受影响
  • 你自己的内容 — 只移除 OpenSpec 标记块，你的内容完整保留

迁移 project.md

旧的 openspec/project.md 需要手动迁移到 config.yaml。可以用这个提示词帮助：

我正在将 OpenSpec 的旧 project.md 迁移到新 config.yaml。

这是我当前的 project.md：
[粘贴 project.md 内容]

请帮我创建 config.yaml，包含：
1. 简洁的 context 部分（会注入到每个规划请求）
2. 按产物分组的 rules（如果内容是特定产物相关的）

省略 AI 模型已经知道的通用最佳实践。

第十八章：常见问题 FAQ

Q1：OpenSpec 是免费的吗？▼

是的，完全免费开源（MIT 许可证）。OpenSpec 是一个 CLI 工具和框架，不收取任何费用。你需要支付的是 AI 编码助手本身的费用。

Q2：推荐使用什么 AI 模型？▼

OpenSpec 效果最好的是高推理能力模型。推荐：

  • Codex 5.5 和 Claude Opus 4.7（规划和实施都很好）
  • Claude Sonnet、GPT-4o 也可以，但规划能力稍弱
  • 保持干净的上下文窗口，在开始实施前清除上下文
Q3：命令不被识别怎么办？▼
  1. 确认已初始化：openspec init
  2. 重新生成技能文件：openspec update
  3. 检查 .claude/skills/ 目录是否存在
  4. 重启 AI 工具以加载新技能
Q4：如何关闭遥测？▼
export OPENSPEC_TELEMETRY=0

或设置标准 DNT 信号：export DO_NOT_TRACK=1

OpenSpec 只收集命令名称和版本号，不收集参数、路径、内容或个人信息。CI 环境下自动禁用。

Q5：什么时候更新变更 vs 开始新变更？▼

更新现有变更：意图相同、范围缩小、基于学习修正方案。

开始新变更：意图根本改变、范围爆炸式增长、原始变更可以独立完成。

原则：更新保留上下文，新变更提供清晰度。

Q6：如何升级 OpenSpec？▼
# 升级包
npm install -g @fission-ai/openspec@latest

Q7：可以用中文写规范吗？▼

可以！在 openspec/config.yaml 中添加语言指令：

context: |
  语言：中文（简体）
  所有产出物必须用简体中文撰写。
Q8：/opsx:verify 是什么？什么时候用？▼

/opsx:verify 从三个维度验证实施是否匹配规范：

  • 完整性：所有任务完成、所有需求已实现、场景已覆盖
  • 正确性：实施匹配规范意图、边界条件已处理
  • 一致性：设计决策反映在代码中、命名约定一致

建议归档前运行。不会阻止归档，但会提前发现问题。

Q9：多个变更能并行吗？▼

可以！每个变更有独立文件夹，互不冲突。使用 /opsx:apply <name> 指定要实施的变更。完成后用 /opsx:bulk-archive 批量归档。

Q10：Delta 规范是什么？▼

Delta 规范只描述变化，而不是重写整个规范。使用三种段落：

  • ## ADDED Requirements — 新增行为，归档时追加到主规范
  • ## MODIFIED Requirements — 变更行为，归档时替换现有需求
  • ## REMOVED Requirements — 废弃行为，归档时从主规范删除

这让两个变更可以同时修改同一个规范文件而不冲突（只要改的是不同的需求）。
