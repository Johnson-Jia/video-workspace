第一章：项目概览

Superpowers 是什么？

Superpowers 是一套完整的 AI 编程方法论系统，以插件形式安装在你的编程 AI 代理（如 Claude Code、Codex CLI、Gemini CLI 等）中。它不是框架、不是库、不是命令行工具 — 它是一组精心设计的技能（Skills），让你的 AI 代理从"随便写写代码"变成"遵循专业软件工程方法论"。

简单说：安装 Superpowers 之后，你的 AI 编程助手会自动：

  • 在写代码之前先跟你讨论需求和设计
  • 严格遵循 TDD（测试驱动开发）流程
  • 按计划分派子代理并行执行任务
  • 完成后自动代码审查和验证

一句话理解

Superpowers = 给你的 AI 编程代理装上"专业软件工程师的方法论大脑"。

第二章：架构与原理

2.1 核心架构

Superpowers 由三个核心组件构成：

Skills（技能）

14 个 Markdown 文件，定义了 AI 代理在各种场景下的行为规范。每个技能包含触发条件、执行步骤、检查清单。

Hooks（钩子）

SessionStart 钩子在会话启动时自动注入 using-superpowers 技能内容，确保 AI 代理从第一刻起就知道如何使用技能系统。

Plugin Config（插件配置）

为每个 AI 代理平台提供标准的插件注册文件（plugin.json），让代理能发现和加载 Superpowers。

2.2 工作原理

会话启动

SessionStart 钩子自动执行，将 using-superpowers 技能内容注入到 AI 代理的系统提示中

Hook → using-superpowers

技能发现

AI 代理了解到有 14 个可用技能，以及每个技能的触发条件。在执行任何任务前，它会检查是否有相关技能

using-superpowers → Skill Tool

技能加载

当检测到适用场景，AI 代理通过 Skill Tool 加载对应技能的完整内容

Skill Tool → 具体技能 SKILL.md

技能执行

AI 代理严格按照技能定义的步骤执行，不跳步、不省略

按 SKILL.md 中的步骤逐一执行

2.4 零依赖设计

为什么零依赖很重要？

Superpowers 的设计哲学是零外部依赖。所有技能都是纯 Markdown 文件，所有逻辑都通过 AI 代理的原生能力执行。这意味着：

  • 安装即用，无需安装 Node.js / Python / 任何运行时
  • 不会与你的项目依赖冲突
  • 维护者拒绝引入第三方依赖的 PR（这是硬性规则）

第五章：技能系统详解

Superpowers 的核心是 14 个技能。每个技能都是一个 Markdown 文件，定义了什么时候触发、按什么步骤执行、检查清单是什么。

技能分类

设计类

brainstorming — 在动手写代码前，通过苏格拉底式提问帮你理清需求

计划类

writing-plans — 将设计拆解为 2-5 分钟的小任务
executing-plans — 按批次执行计划

执行类

subagent-driven-development — 子代理并行执行
dispatching-parallel-agents — 调度独立任务

质量类

test-driven-development — RED-GREEN-REFACTOR
systematic-debugging — 4 阶段根因分析
verification-before-completion — 完成前强制验证

协作类

requesting-code-review — 请求代码审查
receiving-code-review — 接收审查反馈
finishing-a-development-branch — 完成分支

工具类

using-git-worktrees — Git Worktree 隔离
writing-skills — 编写自定义技能
using-superpowers — 入门引导

技能触发机制

重要：技能是强制工作流，不是建议

Superpowers 的技能会在适用场景自动触发。AI 代理在执行任何任务前都会检查是否有相关技能。这不是可选的参考文档，而是必须遵循的方法论。

触发判断遵循"1% 原则"：如果某个技能哪怕只有 1% 的可能适用，AI 代理就必须先加载该技能，确认是否需要执行。

第六章：开发工作流

以下是使用 Superpowers 的典型开发工作流，从想法到交付的完整过程：

Step 1：头脑风暴（brainstorming）

你提出一个想法，AI 代理自动触发头脑风暴技能。通过苏格拉底式提问，帮你澄清需求、探索方案、形成设计文档。

自动触发 — 在任何创造性工作之前

Step 2：创建隔离环境（using-git-worktrees）

设计确认后，AI 代理自动创建一个隔离的 Git Worktree 分支，避免污染主分支。

自动触发 — 设计批准后

Step 3：编写实现计划（writing-plans）

将设计拆解为一系列 2-5 分钟的小任务，每个任务都有明确的文件路径、完整代码和验证步骤。

自动触发 — 设计批准后

Step 4：执行实现（subagent-driven-development）

为每个任务分派一个全新的子代理执行。每完成一个任务后进行两阶段审查（规范合规 + 代码质量）。

自动触发 — 计划就绪后

Step 5：测试驱动开发（test-driven-development）

在实现过程中，严格执行 RED-GREEN-REFACTOR 循环：先写失败测试 → 看到测试失败 → 写最小实现代码 → 看到测试通过。

自动触发 — 实现任何功能时

Step 6：代码审查（requesting-code-review）

任务之间和完成时，自动进行代码审查。按严重级别报告问题，关键问题会阻止继续推进。

自动触发 — 任务完成时

Step 7：完成验证（verification-before-completion）

所有测试通过后，AI 代理强制运行验证命令并确认输出，然后才能声明任务完成。

自动触发 — 声明完成之前

Step 8：完成分支（finishing-a-development-branch）

验证测试通过后，提供选项：合并到主分支 / 创建 PR / 保留分支 / 放弃更改。

自动触发 — 任务全部完成时

你不需要记住这些步骤

Superpowers 的技能会自动触发。你只需要跟 AI 代理正常对话，它会自动在正确的时机执行正确的工作流。上面的流程图只是帮你理解背后的原理。

第七章：核心理念

Superpowers 的设计遵循四个核心理念：

测试驱动开发

先写测试，永远先写测试。任何在测试之前写的代码都会被删除。这不是建议，是强制规则。

系统化 > 临时

流程 > 猜测。遇到 Bug 时不是随意修改代码，而是遵循 4 阶段根因分析流程。

复杂性降低

简洁是首要目标。遵循 YAGNI（你不会需要它）原则，不做过度设计。

证据 > 声明

验证后再宣布成功。"测试通过了"不算成功，AI 代理必须亲自运行验证命令并确认输出。

TDD 不是可选项

在 Superpowers 的工作流中，测试驱动开发是强制执行的。如果你尝试让 AI 代理先写实现代码再写测试，它会拒绝并告诉你"先写测试"。

第八章：技能手册 — 头脑风暴

设计必用

brainstorming

在任何创造性工作（创建功能、构建组件、添加功能、修改行为）之前必须使用。

它做什么？

当你说"我想做一个 XXX 功能"时，AI 代理不会立即开始写代码，而是通过一系列问题帮你：

  • 澄清你真正想要什么
  • 探索不同的实现方案
  • 识别潜在风险和边缘情况
  • 形成清晰的设计文档

使用场景

场景AI 代理的行为
你说"做一个登录页面"触发 brainstorming，问你用什么认证方式、需要社交登录吗、密码策略等
你说"修复这个 Bug"触发 systematic-debugging，先定位根因再修
你说"重构这段代码"触发 brainstorming，确认重构目标和范围

使用示例

你只需要这样说
我想给应用添加一个暗黑模式

AI 代理会自动触发 brainstorming 技能，开始向你提问。

第九章：技能手册 — 测试驱动开发

质量强制

test-driven-development

在实现任何功能或修复任何 Bug 之前触发。强制执行 RED-GREEN-REFACTOR 循环。

TDD 循环

RED：写失败测试
→
看到测试失败
→
GREEN：写最小实现
→
看到测试通过
→
REFACTOR：重构

关键规则

在测试之前写的代码会被删除

如果你不小心先写了实现代码，AI 代理会：

  1. 将实现代码暂存（stash）
  2. 先写测试
  3. 确认测试失败（RED）
  4. 恢复实现代码，确认测试通过（GREEN）
  5. 进行重构

第十章：技能手册 — 系统化调试

质量调试

systematic-debugging

遇到任何 Bug、测试失败或意外行为时触发。在提出修复方案之前必须先完成根因分析。

4 阶段根因分析流程

阶段名称做什么
1复现确认问题可以被稳定复现
2定位通过日志、断点等手段找到根因
3修复针对根因编写修复代码（先写测试！）
4防御添加回归测试，防止问题再次发生

禁止"试试看"式修复

AI 代理不会说"可能是这个原因，我先改改试试"。它必须先证明根因，然后才能修复。

第十一章：技能手册 — 计划编写与执行

计划设计后

writing-plans

将设计文档拆解为具体的实现计划。每个任务 2-5 分钟可完成，包含精确的文件路径、完整代码和验证步骤。

计划的质量标准

一个好的 Superpowers 计划像一份"给热情但缺乏判断力的初级工程师的说明书"：

  • 每个任务都有精确的文件路径
  • 包含完整的代码（不是伪代码或占位符）
  • 每个任务都有验证步骤
  • 任务之间有明确的依赖关系
  • 遵循 YAGNI（你不会需要它）和 DRY（不要重复自己）原则
执行计划后

executing-plans

按批次执行计划，在每个批次完成后设置检查点，等待你的确认再继续。

第十二章：技能手册 — 子代理开发

执行核心

subagent-driven-development

为计划中的每个任务分派一个全新的子代理执行。包含两阶段审查：规范合规性 + 代码质量。

工作方式

分派子代理

主代理为当前任务创建一个全新的子代理，给它完整的任务描述和上下文。

子代理执行

子代理独立完成任务（遵循 TDD、验证等技能）。

第一阶段审查：规范合规

检查子代理的产出是否符合计划中的规格要求。

第二阶段审查：代码质量

检查代码的可维护性、安全性和性能。

继续下一个任务

审查通过后，分派下一个任务的子代理。

Claude 可以自主工作数小时

使用子代理驱动开发，Claude 经常可以自主工作 2 小时以上而不偏离计划。你只需要审批设计和计划，然后让它自己干。

并行调度

执行并行

dispatching-parallel-agents

当有 2 个以上互相独立的任务时，可以同时分派多个子代理并行工作。

并行调度的前提条件：

  • 任务之间没有依赖关系
  • 任务之间没有文件重叠
  • 并行带来的收益值得额外的协调开销

第十三章：技能手册 — 代码审查

协作质量门

requesting-code-review

在完成任务、实现重大功能或合并前触发。验证工作是否满足需求。

审查流程

阶段检查内容
规范合规是否实现了计划中的所有要求？是否有遗漏？
代码质量可维护性、安全性、性能、命名规范
测试覆盖是否有足够的测试？边缘情况是否覆盖？
协作反馈

receiving-code-review

收到代码审查反馈后触发。要求技术严谨和验证，而不是盲目同意和实施。

不是盲目接受反馈

Superpowers 的代码审查反馈处理要求 AI 代理独立思考。如果反馈不明确或技术上有疑问，AI 代理会主动质疑而不是盲目执行。

第十四章：技能手册 — Git Worktree

工具隔离

using-git-worktrees

在开始需要隔离的功能开发或执行实现计划之前触发。创建隔离的 Git Worktree 工作空间。

工作方式

  1. 检测当前是否已在 Worktree 中（避免重复创建）
  2. 优先使用 AI 代理的原生 Worktree 工具（如 Codex 的内置功能）
  3. 回退到 git worktree 命令
  4. 所有 Worktree 创建在 .worktrees/ 目录下（可追踪来源）

v5.1.0 改进

从 v5.1.0 开始，Worktree 创建前会先征得你的同意，不再静默创建。同时修复了多个跨平台兼容性问题。

工具完成

finishing-a-development-branch

所有任务完成、测试通过后触发。提供四个选项：

选项说明
合并到主分支自动 merge 并清理 Worktree
创建 Pull Request推送分支并创建 PR
保留分支保留当前状态，稍后处理
放弃更改删除分支和 Worktree

第十五章：技能手册 — 完成验证

质量门禁

verification-before-completion

在声明任务完成、修复成功或测试通过之前触发。证据先于断言，永远如此。

验证清单

  • 所有测试命令已执行并确认输出（不是"应该通过了"）
  • 功能在真实环境中验证（不是单元测试通过就完事）
  • 没有引入回归（其他功能仍然正常工作）
  • 边缘情况已测试

"测试通过了" 不等于 "完成了"

AI 代理不能说"我运行了测试，应该没问题"。它必须实际运行验证命令，查看输出，并确认结果符合预期。

元技能扩展

writing-skills

第十七章：贡献指南

如果你想为 Superpowers 项目贡献代码，请仔细阅读以下规则：

PR 拒绝率 94%

这个项目的 PR 拒绝率极高，主要原因是 AI 代理提交的低质量 PR。提交前请务必确认你的 PR 符合以下所有要求。

17.1 贡献流程

  1. Fork 仓库
  2. 切换到 dev 分支
  3. 创建你的工作分支
  4. 遵循 writing-skills 技能开发和测试变更
  5. 填写完整的 PR 模板

17.2 不会被接受的内容

类型说明
第三方依赖Superpowers 是零依赖项目，不引入任何外部依赖
技能"合规"重写不能为了"符合文档规范"而重写技能内容
项目特定配置只服务于特定项目/团队的配置不属于核心
批量 PR不要一次性针对多个 Issue 提交 PR
推测性修复"可能有问题"不是问题陈述，必须有真实场景
领域特定技能只服务特定领域的技能应发布为独立插件
捏造内容包含虚假声明或捏造功能描述的 PR 会被立即关闭

17.3 PR 必须包含的内容

  • 完整的 PR 模板（每个部分都填写真实内容）
  • 搜索过已有的 PR（开放的和关闭的）
  • 描述了真实存在的问题（不是理论上的）
  • 确认变更属于核心（不是领域特定的）
  • 人工审查了完整的 diff

第十九章：常见问题 FAQ

Q1：Superpowers 是免费的吗？▼

是的，完全免费且开源。 Superpowers 采用 MIT 协议发布。你可以在任何项目中免费使用。如果觉得有帮助，可以考虑 赞助作者。

Q2：安装后 AI 代理直接写代码，没有先讨论需求？▼

这说明 Superpowers 没有正确加载。请检查：

  1. 确认安装命令执行成功
  2. 重启 AI 代理（完全退出再打开）
  3. 使用验收测试验证：我们来做一个 React 待办列表
  4. 如果还不行，检查插件市场是否正常连接
Q3：我能在多个 AI 代理中使用吗？▼

可以。 Superpowers 支持 Claude Code、Codex CLI/App、Gemini CLI、OpenCode、Cursor、GitHub Copilot CLI 和 Factory Droid。但每个代理需要单独安装。

Q4：Superpowers 会影响 AI 代理的性能吗？▼

影响很小。SessionStart 钩子会在启动时注入 using-superpowers 技能内容（增加系统提示长度），但每个具体技能只在需要时才加载。零依赖设计意味着没有运行时开销。

Q5：可以禁用某个技能吗？▼

目前没有官方的禁用单个技能的机制。如果你不想使用某个技能（比如不想用 TDD），可以在对话中明确告诉 AI 代理跳过。但注意：技能是设计为强制工作流的，绕过它们可能会降低输出质量。

Q6：如何更新到最新版本？▼

更新方式取决于你的 AI 代理：

  • Claude Code：通常自动更新。或执行 /plugin update superpowers
  • Gemini CLI：gemini extensions update superpowers
  • Codex：通过插件界面检查更新
  • OpenCode：更新 opencode.json 中的版本引用
Q7：Superpowers 和普通 AI 编程有什么区别？▼
对比项没有 Superpowers有 Superpowers
写代码前直接写先讨论需求和设计
测试写完代码再补测试先写测试，再写实现
Bug 修复猜测原因，改改试试系统化根因分析
任务执行一个人从头做到尾分派子代理并行执行
代码审查没有自动审查两阶段自动审查
完成标准"应该没问题"必须运行验证命令确认
Q8：我需要什么编程水平才能使用？▼

Superpowers 本身不需要你写代码，但它理解的概念（测试、分支、代码审查）是软件工程基础知识。如果你：

  • 会使用 AI 编程代理 → 就能使用 Superpowers
  • 了解 Git 基础 → 能更好地利用 Worktree 和分支管理技能
  • 了解 TDD 概念 → 能更好地理解 AI 代理的工作方式
Q9：技能里的 "human partner" 是什么意思？▼

human partner（人类伙伴）是 Superpowers 对用户的称呼。这是项目 deliberate（深思熟虑）的用词选择，代表 AI 代理和用户之间的协作关系，而不是简单的"服务者-被服务者"关系。在技能内容中不应替换为 "user" 或其他词。

Q10：如何获取帮助或报告问题？▼
  • GitHub Issues：github.com/obra/superpowers/issues
  • Discord 社区：加入 Discord 获取社区支持和交流
  • 版本公告：订阅更新通知
