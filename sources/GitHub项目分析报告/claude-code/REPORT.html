1.1 什么是 Claude Code

Claude Code 是 Anthropic 公司推出的官方 Agentic 编码 CLI 工具，直接运行在终端中。它不是一个简单的代码补全工具，而是一个具有高度自主性的 AI 编程代理——能理解代码库、编辑文件、运行命令、执行测试、创建 PR，甚至编排多个 Agent 并行工作。

第二章：源码泄露事件

2.1 泄露方式

Source Map 泄露

Anthropic 在 npm 发布 @anthropic-ai/claude-code 包时，未排除 source map 文件（cli.js.map）。Source map 中的 sourcesContent 字段存储了编译前的完整 TypeScript 源码。

cli.js.map → 提取 sourcesContent → 1,987 个 .ts/.tsx 文件 → 完整项目树

2.2 泄露范围

1,987 个 TypeScript/TSX 文件 ~513,681 行代码 ~30 MB 数据大小 ~100% 完整度

2.3 行业影响

维度影响
媒体评价VentureBeat: "竞争对手获得了 agentic 编码工具的蓝图"
安全研究社区发现 Bash 工具有沙箱和权限控制，但需进一步审计
行业共识AI 编码工具的护城河不是模型，而是 harness（编排层）
供应链风险泄露同期有人发布恶意 axios npm 包

第五章：三层功能门控体系

这是 Claude Code 架构中最精妙的设计——外部发布版是经过三层过滤的精简版。

第一层：编译时 feature() — ~50 个开关

Bun bundler DCE 死代码消除。外部版构建时直接删除被关闭功能的代码，不可逆向恢复。

第二层：USER_TYPE 硬编码

'ant'（Anthropic 内部）vs 'external'（外部用户）。200+ 处条件检查，控制工具/命令/UI 可见性。

第三层：GrowthBook 远程 A/B 测试

tengu_* 前缀开关，运行时动态控制。支持灰度发布、Kill Switch、用户分群。

第六章：七大隐藏功能深度解析

6.1 BUDDY — AI 电子宠物系统

编译开关: feature('BUDDY') | 源码: src/buddy/ | 计划上线: 2026 年 4 月

终端里的拓麻歌子——一个完整的虚拟宠物系统：

🦆鸭子 🪿鹅 🫧果冻 🐱猫 🐉龙 🐙章鱼 🦉猫头鹰 🐧企鹅 🐢乌龟 🐌蜗牛 👻幽灵 🦎六角恐龙 🦫水豚 🌵仙人掌 🤖机器人 🐰兔子 🍄蘑菇 🐈胖猫

稀有度体系: Common (60%) → Uncommon (25%) → Rare (10%) → Epic (4%) → Legendary (1%)，独立 1% 闪光概率

防作弊确定性生成: 用户 UUID + 盐值 'friend-2026-401' → FNV-1a 哈希 → Mulberry32 PRNG，每人固定一只宠物

外观系统: 6 种眼睛 + 8 种帽子（皇冠、巫师帽、光环等）

交互: /buddy pet（爱心动画）、/buddy hatch（孵化）、/buddy card（卡片）

动画: 500ms 帧率 ASCII 精灵动画，窄终端退化为 =·ω·=

6.2 KAIROS — 永不关机的 Claude

编译开关: feature('KAIROS') | 源码: src/assistant/, src/proactive/, src/services/autoDream/

最复杂的隐藏功能——将 Claude 从"一次性对话工具"变为"持久 AI 助手"。

  • 跨会话持久运行: 关闭终端后仍在后台运行
  • 自动做梦（Dream）: 四阶段记忆整合（Orient → Gather → Consolidate → Prune）
  • 每日日志: <autoMemPath>/logs/YYYY/MM/YYYY-MM-DD.md
  • 主动模式: 没人说话时自己找活干，周期性 <tick> 提示驱动
  • 持久 Cron: permanent: true 任务不受 7 天过期限制

做梦三层门控: 时间门控（24h）→ 会话门控（5+ 新会话）→ 锁门控（PID 存活检查）

激活五层检查: feature('KAIROS') → settings.assistant: true → 目录信任检查 → tengu_kairos → setKairosActive(true)

6.3 ULTRAPLAN — 云端深度规划

编译开关: feature('ULTRAPLAN') | 完全内部限定: "external" === 'ant' 永远为 false

将复杂任务发送到云端 Opus 模型独立研究最长 30 分钟。

/ultraplan <prompt>
→ 创建远程 CCR 会话
→ Opus 模型独立研究（30 分钟超时）
→ 后台轮询等待
→ 浏览器查看/修改方案
→ 批准执行或传送回本地

传送系统（Teleport）: Git Bundle 打包代码上下文，本地 ↔ 远程双向传输。

6.4 Coordinator — 多 Agent 编排

编译开关: feature('COORDINATOR_MODE') | 源码: src/coordinator/

  • 角色分离: Coordinator 只有 3 个工具（Agent、SendMessage、TaskStop）
  • Worker 拥有完整工具集: 在独立子进程中运行
  • 核心铁律: 系统提示明确"禁止甩锅式委派"——不能把不清楚的需求直接丢给 Worker
  • 共享任务列表: ~/.claude/tasks/，Coordinator 和 Worker 共同读写

6.5 Bridge — 远程遥控终端

编译开关: feature('BRIDGE_MODE') | 源码: src/bridge/（33 个文件）

  • 两代协议: v1（环境层轮询）和 v2（SSE + CCR 直连）
  • 双向控制: 网页输入 → 本地执行，本地输出 → 网页显示
  • 安全机制: OAuth + Worker JWT + 设备信任令牌
  • 崩溃恢复: bridge-pointer.json 支持断线重连

6.6 Voice — 语音交互模式

编译开关: feature('VOICE_MODE')，远程开关: tengu_cobalt_frost（Nova 3 语音识别）

第十章：总结与洞察

10.1 核心发现

发现一：不是简单的 CLI 包装器

53 个工具、87 个命令、148 个 UI 组件、50 个编译开关的复杂 agentic 编码系统。51 万行源码构成了一个精心设计的 Agent Harness。

发现二：外部版是精简版

通过三层门控（编译开关 → 用户类型 → 远程 A/B 测试）精确控制功能可见性。大量高级功能对外部用户不可用。

发现三：架构设计精良

React + Ink 终端 UI、MCP 协议集成、GrowthBook 远程配置、OAuth/JWT 安全体系、多 Agent 编排、自动记忆整合等，代表了当前 AI 编码工具的最高工程水平。

发现四：真正的护城河是 Harness

源码泄露揭示了 AI 编码工具的真正护城河：不是底层模型，而是编排层（harness）的工程复杂度——上下文管理、工具编排、权限控制、记忆系统、多 Agent 协作。

10.2 架构亮点

  • 确定性宠物生成: FNV-1a + Mulberry32 PRNG，防作弊设计思路值得学习
  • 做梦机制: 四阶段记忆整合，三层门控防资源浪费
  • 禁止甩锅式委派: Coordinator 系统提示中的设计哲学
  • 三层功能门控: 编译时 + 硬编码 + 远程配置的组合拳
  • Prompt Cache: 成本优化的工程实践
